87  // begin
88  auto begin() {
89  /*
90  * Why the incrementation (++)?
91  * Normally a tree_iterator<T> is constructed by taking a reference to a
92  * tree<T> object. The iterator constructed this way points to the
93  * end(). This end() status is the same for both iterator and
94  * reverse_iterator. Incrementing an iterator at the end() will take it
95  * to the first element (this is why we (++) here). Decrementing an
96  * iterator at the end() will take it to the last element (end is one
97  * step past last). std::reverse_iterator will take an underlying
98  * iterator as argument and translate a (++) operator on itself in a
99  * (--) operation on the underlying iterator and vice versa.
100  *
101  * Why we don't increment (++) the reverse iterator as we do with
102  * iterator? It would make sense: when constructed the iterator is at
103  * the end() (which is same for both iterator and reverse_iterator)
104  * incrementing will result in an actual decrementing taking the
105  * iterator to the last element (that is reverse_iterator's fist
106  * element). We don't need to do this because a reverse_iterator will
107  * be decremented before dereferenced (every time).
108  * See "http://en.cppreference.com/w/cpp/iterator/reverse_iterator/operator*".
109  * This seems like a waste of resources to me but this is how it works.
110  *
111  * Also the reason reverse end iterator are incremented should be clear
112  * now. In general reverse_iterator == iterator - 1
113  */
114  return ++iterator<Algorithm>(*this);
115  }
