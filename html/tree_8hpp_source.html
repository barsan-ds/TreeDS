145  template<typename A = Algorithm>
146  iterator<A> begin() {
147  /*
148  * Why the incrementation (++)?
149  * Normally a tree_iterator<T> is constructed by taking a reference to a
150  * tree<T> object. The iterator constructed this way points to the
151  * end(). This end() status is the same for both iterator and
152  * reverse_iterator. Incrementing an iterator at the end() will take it
153  * to the first element (this is why we (++) here). Decrementing an
154  * iterator at the end() will take it to the last element (end is one
155  * step past last). std::reverse_iterator will take an underlying
156  * iterator as argument and translate a (++) operator on itself in a
157  * (--) operation on the underlying iterator and vice versa.
158  *
159  * Why we don't increment (++) the reverse iterator as we do with
160  * iterator? It would make sense: when constructed the iterator is at
161  * the end() (which is same for both iterator and reverse_iterator)
162  * incrementing will result in an actual decrementing taking the
163  * iterator to the last element (that is reverse_iterator's fist
164  * element). We don't need to do this because a reverse_iterator will
165  * be decremented before dereferenced (every time).
166  * See "http://en.cppreference.com/w/cpp/iterator/reverse_iterator/operator*".
167  * This seems like a waste of resources to me but this is how it works.
168  *
169  * Also the reason reverse end iterator are incremented should be clear
170  * now. In general reverse_iterator == iterator - 1
171  */
172  return ++iterator<A>(this);
173  }
